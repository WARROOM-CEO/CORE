# สร้างปลั๊กอิน Cowork

สร้างปลั๊กอินใหม่ตั้งแต่ต้นผ่านการสนทนาแบบมีคำแนะนำ นำผู้ใช้ผ่านขั้นตอนการค้นพบ การวางแผน การออกแบบ การสร้าง และการแพ็กเกจ — ส่งมอบไฟล์ `.plugin` พร้อมติดตั้งเมื่อเสร็จ

## ภาพรวม

ปลั๊กอินคือไดเรกทอรีแบบ self-contained ที่ขยายความสามารถของ Claude ด้วย skill, agent, hook และการเชื่อมต่อ MCP เซิร์ฟเวอร์ Skill นี้มีสถาปัตยกรรมปลั๊กอินทั้งหมดและขั้นตอนการทำงาน 5 ขั้นตอนสำหรับการสร้างปลั๊กอินแบบสนทนา

กระบวนการ:

1. **การค้นพบ** — ทำความเข้าใจสิ่งที่ผู้ใช้ต้องการสร้าง
2. **การวางแผนส่วนประกอบ** — กำหนดว่าต้องการส่วนประกอบประเภทใดบ้าง
3. **การออกแบบและคำถามชี้แจง** — ระบุรายละเอียดแต่ละส่วนประกอบ
4. **การสร้าง** — สร้างไฟล์ปลั๊กอินทั้งหมด
5. **การตรวจสอบและแพ็กเกจ** — ส่งมอบไฟล์ `.plugin`

> **ผลลัพธ์ที่ไม่เป็นเทคนิค**: การสนทนากับผู้ใช้ทั้งหมดใช้ภาษาเข้าใจง่าย ไม่เปิดเผยรายละเอียดการสร้างเช่น path ไฟล์ โครงสร้างไดเรกทอรี หรือฟิลด์ schema เว้นแต่ผู้ใช้จะถาม อธิบายทุกอย่างในแง่ของสิ่งที่ปลั๊กอินจะทำ

## สถาปัตยกรรมปลั๊กอิน

### โครงสร้างไดเรกทอรี

ปลั๊กอินทุกตัวมีโครงสร้างดังนี้:

```
plugin-name/
├── .claude-plugin/
│   └── plugin.json           # จำเป็น: manifest ปลั๊กอิน
├── skills/                   # Skill (ไดเรกทอรีย่อยที่มี SKILL.md)
│   └── skill-name/
│       ├── SKILL.md
│       └── references/
├── agents/                   # คำจำกัดความ subagent (ไฟล์ .md)
├── .mcp.json                 # คำจำกัดความ MCP เซิร์ฟเวอร์
└── README.md                 # เอกสารปลั๊กอิน
```

> **รูปแบบ `commands/` แบบเดิม**: ปลั๊กอินเก่าอาจมีไดเรกทอรี `commands/` ที่มี slash command ไฟล์เดียว (.md) รูปแบบนี้ยังใช้ได้ แต่ปลั๊กอินใหม่ควรใช้ `skills/*/SKILL.md` แทน — UI ของ Cowork แสดงทั้งสองเป็นหมวด "Skills" เดียว และรูปแบบ skills รองรับ progressive disclosure ผ่าน `references/`

**กฎ:**

- `.claude-plugin/plugin.json` จำเป็นเสมอ
- ไดเรกทอรีส่วนประกอบ (`skills/`, `agents/`) อยู่ที่ root ของปลั๊กอิน ไม่ใช่ภายใน `.claude-plugin/`
- สร้างไดเรกทอรีเฉพาะส่วนประกอบที่ปลั๊กอินใช้จริง
- ใช้ kebab-case สำหรับชื่อไดเรกทอรีและไฟล์ทั้งหมด

### plugin.json Manifest

อยู่ที่ `.claude-plugin/plugin.json` ฟิลด์ที่จำเป็นขั้นต่ำคือ `name`

```json
{
  "name": "plugin-name",
  "version": "0.1.0",
  "description": "คำอธิบายสั้นๆ เกี่ยวกับวัตถุประสงค์ของปลั๊กอิน",
  "author": {
    "name": "ชื่อผู้สร้าง"
  }
}
```

**กฎการตั้งชื่อ:** kebab-case, ตัวพิมพ์เล็กพร้อมขีดกลาง, ไม่มีช่องว่างหรืออักขระพิเศษ
**เวอร์ชัน:** รูปแบบ semver (MAJOR.MINOR.PATCH) เริ่มที่ `0.1.0`

ฟิลด์เพิ่มเติม: `homepage`, `repository`, `license`, `keywords`

สามารถระบุ path ส่วนประกอบที่กำหนดเอง (เสริมไม่ใช่แทนที่การค้นหาอัตโนมัติ):

```json
{
  "commands": "./custom-commands",
  "agents": ["./agents", "./specialized-agents"],
  "hooks": "./config/hooks.json",
  "mcpServers": "./.mcp.json"
}
```

### Schema ส่วนประกอบ

Schema ละเอียดสำหรับส่วนประกอบแต่ละประเภทอยู่ใน `references/component-schemas.md` สรุป:

| ส่วนประกอบ | ตำแหน่ง | รูปแบบ |
| --- | --- | --- |
| Skill | `skills/*/SKILL.md` | Markdown + YAML frontmatter |
| MCP เซิร์ฟเวอร์ | `.mcp.json` | JSON |
| Agent (ไม่ค่อยใช้ใน Cowork) | `agents/*.md` | Markdown + YAML frontmatter |
| Hook (ใช้น้อยมากใน Cowork) | `hooks/hooks.json` | JSON |
| Command (แบบเดิม) | `commands/*.md` | Markdown + YAML frontmatter |

Schema นี้ใช้ร่วมกับระบบปลั๊กอินของ Claude Code แต่คุณกำลังสร้างปลั๊กอินสำหรับ Claude Cowork ซึ่งเป็นแอปเดสก์ท็อปสำหรับงาน knowledge work
ผู้ใช้ Cowork มักจะพบว่า skill มีประโยชน์มากที่สุด **สร้างโครงร่างปลั๊กอินใหม่ด้วย `skills/*/SKILL.md` — ไม่ต้องสร้าง `commands/` เว้นแต่ผู้ใช้จะต้องการรูปแบบไฟล์เดียวแบบเดิมโดยเฉพาะ**

### ปลั๊กอินที่ปรับแต่งได้ด้วย placeholder `~~`

> **ไม่ต้องใช้หรือถามเกี่ยวกับรูปแบบนี้โดยค่าเริ่มต้น** แนะนำ placeholder `~~` เฉพาะเมื่อผู้ใช้ระบุชัดเจนว่าต้องการให้คนนอกองค์กรใช้ปลั๊กอิน
> สามารถกล่าวถึงตัวเลือกนี้หากดูเหมือนว่าผู้ใช้ต้องการเผยแพร่ปลั๊กอินภายนอก แต่ไม่ต้องถามเชิงรุกด้วย AskUserQuestion

เมื่อปลั๊กอินมีจุดประสงค์เพื่อแบ่งปันกับผู้อื่นนอกบริษัท อาจมีส่วนที่ต้องปรับเปลี่ยนตามผู้ใช้แต่ละคน
อาจต้องอ้างอิงเครื่องมือภายนอกตามหมวดหมู่แทนผลิตภัณฑ์เฉพาะ (เช่น "เครื่องมือจัดการโปรเจกต์" แทน "Jira")
เมื่อต้องการแบ่งปัน ใช้ภาษาทั่วไปและทำเครื่องหมายจุดที่ต้องปรับแต่งด้วยตัวหนอน (tilde) สองตัว เช่น `สร้าง issue ใน ~~เครื่องมือจัดการโปรเจกต์`
หากใช้หมวดหมู่เครื่องมือ สร้างไฟล์ `CONNECTORS.md` ที่ root ของปลั๊กอินเพื่ออธิบาย:

```markdown
# Connectors

## การอ้างอิงเครื่องมือทำงานอย่างไร

ไฟล์ปลั๊กอินใช้ `~~หมวดหมู่` เป็น placeholder สำหรับเครื่องมือที่ผู้ใช้เชื่อมต่อในหมวดหมู่นั้น ปลั๊กอินเป็นแบบ tool-agnostic — อธิบายเวิร์กโฟลว์ในแง่ของหมวดหมู่แทนผลิตภัณฑ์เฉพาะ

## Connectors สำหรับปลั๊กอินนี้

| หมวดหมู่ | Placeholder | ตัวเลือก |
| --- | --- | --- |
| แชท | `~~แชท` | Slack, Microsoft Teams, Discord |
| เครื่องมือจัดการโปรเจกต์ | `~~เครื่องมือจัดการโปรเจกต์` | Linear, Asana, Jira |
```

### ตัวแปร ${CLAUDE_PLUGIN_ROOT}

ใช้ `${CLAUDE_PLUGIN_ROOT}` สำหรับการอ้างอิง path ภายในปลั๊กอินทั้งหมดใน hook และกำหนดค่า MCP ห้ามใช้ absolute path แบบ hardcode

## ขั้นตอนการทำงานแบบมีคำแนะนำ

เมื่อถามผู้ใช้ ให้ใช้ AskUserQuestion ไม่ควรสันนิษฐานว่าค่าเริ่มต้น "มาตรฐานอุตสาหกรรม" ถูกต้อง หมายเหตุ: AskUserQuestion จะมีปุ่มข้ามและช่องข้อความอิสระเสมอ ดังนั้นไม่ต้องเพิ่มตัวเลือก "ไม่มี" หรือ "อื่นๆ"

### ขั้นที่ 1: การค้นพบ

**เป้าหมาย**: ทำความเข้าใจสิ่งที่ผู้ใช้ต้องการสร้างและทำไม

ถาม (เฉพาะสิ่งที่ไม่ชัดเจน — ข้ามคำถามหากคำขอเริ่มต้นของผู้ใช้ตอบไปแล้ว):

- ปลั๊กอินนี้ควรทำอะไร? แก้ปัญหาอะไร?
- ใครจะใช้และในบริบทใด?
- ต้องเชื่อมต่อกับเครื่องมือหรือบริการภายนอกหรือไม่?
- มีปลั๊กอินหรือเวิร์กโฟลว์ที่คล้ายกันเพื่ออ้างอิงหรือไม่?

สรุปความเข้าใจและยืนยันก่อนดำเนินการต่อ

**ผลลัพธ์**: คำชี้แจงที่ชัดเจนเกี่ยวกับวัตถุประสงค์และขอบเขตของปลั๊กอิน

### ขั้นที่ 2: การวางแผนส่วนประกอบ

**เป้าหมาย**: กำหนดว่าปลั๊กอินต้องการส่วนประกอบประเภทใดบ้าง

จากคำตอบขั้นตอนค้นพบ กำหนด:

- **Skill** — ต้องการความรู้เฉพาะทางที่ Claude ควรโหลดตามต้องการ หรือการกระทำที่ผู้ใช้เริ่มต้น? (ความเชี่ยวชาญเฉพาะด้าน schema อ้างอิง คู่มือเวิร์กโฟลว์ การ deploy/กำหนดค่า/วิเคราะห์/ตรวจสอบ)
- **MCP เซิร์ฟเวอร์** — ต้องการเชื่อมต่อบริการภายนอก? (ฐานข้อมูล API เครื่องมือ SaaS)
- **Agent (ไม่ค่อยใช้)** — มีงานหลายขั้นตอนแบบอัตโนมัติ? (การตรวจสอบ การสร้าง การวิเคราะห์)
- **Hook (ใช้น้อยมาก)** — ควรมีสิ่งที่เกิดขึ้นอัตโนมัติเมื่อมีเหตุการณ์บางอย่าง? (บังคับใช้นโยบาย โหลดบริบท ตรวจสอบการดำเนินงาน)

นำเสนอตารางแผนส่วนประกอบ รวมถึงประเภทที่ตัดสินใจไม่สร้าง:

```
| ส่วนประกอบ | จำนวน | วัตถุประสงค์ |
|-----------|-------|---------|
| Skill    | 3     | ความรู้เฉพาะด้าน X, /do-thing, /check-thing |
| Agent    | 0     | ไม่จำเป็น |
| Hook     | 1     | ตรวจสอบการเขียน |
| MCP      | 1     | เชื่อมต่อกับบริการ Y |
```

ขอยืนยันหรือปรับเปลี่ยนจากผู้ใช้ก่อนดำเนินการต่อ

**ผลลัพธ์**: รายการส่วนประกอบที่ยืนยันแล้วที่จะสร้าง

### ขั้นที่ 3: การออกแบบและคำถามชี้แจง

**เป้าหมาย**: ระบุรายละเอียดแต่ละส่วนประกอบ แก้ไขความคลุมเครือทั้งหมดก่อนการสร้าง

สำหรับส่วนประกอบแต่ละประเภทในแผน ถามคำถามออกแบบเฉพาะเจาะจง นำเสนอคำถามแบ่งตามประเภทส่วนประกอบ รอคำตอบก่อนดำเนินการต่อ

**Skill:**

- คำถามหรือคำสั่งใดของผู้ใช้ควรเรียกใช้ skill นี้?
- ครอบคลุมด้านความรู้ใด?
- ควรมีไฟล์อ้างอิงสำหรับเนื้อหาละเอียด?
- หาก skill เป็นการกระทำที่ผู้ใช้เริ่มต้น: รับ argument อะไรบ้าง และต้องใช้เครื่องมือใด? (Read, Write, Bash, Grep เป็นต้น)

**Agent:**

- แต่ละ agent ควรทำงานเชิงรุกหรือเฉพาะเมื่อร้องขอ?
- ต้องใช้เครื่องมือใด?
- รูปแบบผลลัพธ์ควรเป็นอย่างไร?

**Hook:**

- เหตุการณ์ใด? (PreToolUse, PostToolUse, Stop, SessionStart เป็นต้น)
- พฤติกรรมอะไร — ตรวจสอบ บล็อก แก้ไข เพิ่มบริบท?
- แบบ prompt (ขับเคลื่อนด้วย LLM) หรือแบบ command (script ที่แน่นอน)?

**MCP เซิร์ฟเวอร์:**

- ประเภทเซิร์ฟเวอร์? (stdio สำหรับ local, SSE สำหรับ hosted ที่มี OAuth, HTTP สำหรับ REST API)
- วิธีการยืนยันตัวตน?
- ควรเปิดให้ใช้เครื่องมือใด?

หากผู้ใช้บอกว่า "ตามที่คุณคิดว่าดีที่สุด" ให้เสนอคำแนะนำเฉพาะเจาะจงและขอยืนยันชัดเจน

**ผลลัพธ์**: ข้อกำหนดละเอียดสำหรับทุกส่วนประกอบ

### ขั้นที่ 4: การสร้าง

**เป้าหมาย**: สร้างไฟล์ปลั๊กอินทั้งหมดตามแนวปฏิบัติที่ดี

**ลำดับการดำเนินการ:**

1. สร้างโครงสร้างไดเรกทอรีปลั๊กอิน
2. สร้าง manifest `plugin.json`
3. สร้างแต่ละส่วนประกอบ (ดู `references/component-schemas.md` สำหรับรูปแบบที่แน่นอน)
4. สร้าง `README.md` เอกสารปลั๊กอิน

**แนวทางการสร้าง:**

- **Skill** ใช้ progressive disclosure: SKILL.md body กระชับ (ไม่เกิน 3,000 คำ) เนื้อหาละเอียดใน `references/` คำอธิบายใน frontmatter ต้องเป็นบุรุษที่สามพร้อมวลีทริกเกอร์เฉพาะ body ของ skill เป็นคำสั่งสำหรับ Claude ไม่ใช่ข้อความถึงผู้ใช้ — เขียนเป็นคำสั่งเกี่ยวกับสิ่งที่ต้องทำ
- **Agent** ต้องมีคำอธิบายพร้อมบล็อก `<example>` แสดงเงื่อนไขทริกเกอร์ บวก system prompt ใน markdown body
- **Hook** กำหนดค่าใน `hooks/hooks.json` ใช้ `${CLAUDE_PLUGIN_ROOT}` สำหรับ path script แนะนำ hook แบบ prompt สำหรับ logic ซับซ้อน
- **กำหนดค่า MCP** อยู่ใน `.mcp.json` ที่ root ปลั๊กอิน ใช้ `${CLAUDE_PLUGIN_ROOT}` สำหรับ path เซิร์ฟเวอร์ local บันทึก env var ที่จำเป็นใน README

### ขั้นที่ 5: การตรวจสอบและแพ็กเกจ

**เป้าหมาย**: ส่งมอบปลั๊กอินที่เสร็จสมบูรณ์

1. สรุปสิ่งที่สร้าง — ระบุแต่ละส่วนประกอบและวัตถุประสงค์
2. ถามว่าผู้ใช้ต้องการปรับเปลี่ยนอะไรไหม
3. รัน `claude plugin validate <path-to-plugin-json>`; แก้ไขข้อผิดพลาดและคำเตือน
4. แพ็กเกจเป็นไฟล์ `.plugin`:

```bash
cd /path/to/plugin-dir && zip -r /tmp/plugin-name.plugin . -x "*.DS_Store" && cp /tmp/plugin-name.plugin /path/to/outputs/plugin-name.plugin
```

> **สำคัญ**: สร้าง zip ใน `/tmp/` ก่อนเสมอ จากนั้นคัดลอกไปยังโฟลเดอร์ outputs การเขียนตรงไปยังโฟลเดอร์ outputs อาจล้มเหลวเนื่องจากสิทธิ์

> **การตั้งชื่อ**: ใช้ชื่อปลั๊กอินจาก `plugin.json` สำหรับไฟล์ `.plugin` (เช่น ถ้าชื่อ `code-reviewer` ให้ output เป็น `code-reviewer.plugin`)

ไฟล์ `.plugin` จะปรากฏในแชทเป็น rich preview ที่ผู้ใช้สามารถเรียกดูไฟล์และยอมรับปลั๊กอินโดยกดปุ่ม

## แนวปฏิบัติที่ดี

- **เริ่มเล็ก**: เริ่มด้วยชุดส่วนประกอบขั้นต่ำที่จำเป็น ปลั๊กอินที่มี skill เดียวที่ดีมีประโยชน์มากกว่าปลั๊กอินที่มีห้า skill ที่ทำได้ครึ่งๆ กลางๆ
- **Progressive disclosure สำหรับ skill**: ความรู้หลักใน SKILL.md, เอกสารอ้างอิงละเอียดใน `references/`, ตัวอย่างการทำงานใน `examples/`
- **วลีทริกเกอร์ที่ชัดเจน**: คำอธิบาย skill ควรรวมวลีเฉพาะที่ผู้ใช้จะพูด คำอธิบาย agent ควรมีบล็อก `<example>`
- **Skill มีไว้สำหรับ Claude**: เขียนเนื้อหา skill body เป็นคำสั่งสำหรับ Claude ไม่ใช่เอกสารสำหรับผู้ใช้อ่าน
- **รูปแบบการเขียนเป็นคำสั่ง**: ใช้คำสั่งที่ขึ้นต้นด้วยกริยาใน skill ("แยกวิเคราะห์ไฟล์กำหนดค่า" ไม่ใช่ "คุณควรแยกวิเคราะห์ไฟล์กำหนดค่า")
- **Portability**: ใช้ `${CLAUDE_PLUGIN_ROOT}` สำหรับ path ภายในปลั๊กอินเสมอ ห้ามใช้ path แบบ hardcode
- **ความปลอดภัย**: ใช้ environment variable สำหรับ credential, HTTPS สำหรับเซิร์ฟเวอร์ระยะไกล, สิทธิ์เข้าถึงเครื่องมือน้อยที่สุด
