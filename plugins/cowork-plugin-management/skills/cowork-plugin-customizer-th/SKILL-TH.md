# การปรับแต่งปลั๊กอิน Cowork

ปรับแต่งปลั๊กอินสำหรับองค์กรเฉพาะ — ไม่ว่าจะเป็นการตั้งค่าเทมเพลตปลั๊กอินทั่วไปครั้งแรก หรือการปรับแต่งปลั๊กอินที่กำหนดค่าไว้แล้ว

> **การค้นหาปลั๊กอิน**: เพื่อค้นหาไฟล์ต้นฉบับของปลั๊กอิน ให้รัน `find mnt/.local-plugins mnt/.plugins -type d -name "*<plugin-name>*"` เพื่อหาตำแหน่งไดเรกทอรีปลั๊กอิน จากนั้นอ่านไฟล์เพื่อทำความเข้าใจโครงสร้างก่อนทำการเปลี่ยนแปลง หากไม่พบไดเรกทอรีปลั๊กอิน ผู้ใช้อาจใช้งานอยู่ใน remote container ให้หยุดและแจ้งว่า: "การปรับแต่งปลั๊กอินใช้ได้เฉพาะในโหมด Cowork ของแอปเดสก์ท็อปเท่านั้น"

## การกำหนดโหมดการปรับแต่ง

หลังจากค้นหาปลั๊กอินแล้ว ตรวจสอบ placeholder ที่ขึ้นต้นด้วย `~~`: `grep -rn '~~\w' /path/to/plugin --include='*.md' --include='*.json'`

> **กฎเริ่มต้น**: หากมี placeholder `~~` อยู่ ให้เริ่มต้นด้วย **การตั้งค่าปลั๊กอินทั่วไป** เว้นแต่ผู้ใช้จะระบุชัดเจนว่าต้องการปรับแต่งส่วนใดส่วนหนึ่งโดยเฉพาะ

**1. การตั้งค่าปลั๊กอินทั่วไป** — ปลั๊กอินมี placeholder ที่ขึ้นต้นด้วย `~~` ซึ่งเป็นจุดปรับแต่งในเทมเพลตที่ต้องแทนที่ด้วยค่าจริง (เช่น `~~Jira` → `Asana`, `~~your-team-channel` → `#engineering`)

**2. การปรับแต่งเฉพาะส่วน** — ไม่มี placeholder `~~` และผู้ใช้ขอปรับแต่งส่วนเฉพาะของปลั๊กอิน (เช่น "ปรับแต่ง connector", "อัปเดต skill standup", "เปลี่ยนเครื่องมือจัดการ ticket") อ่านไฟล์ปลั๊กอินเพื่อหาส่วนที่เกี่ยวข้องและมุ่งเน้นเฉพาะส่วนนั้น ไม่ต้องสแกนทั้งปลั๊กอินหรือแสดงรายการปรับแต่งที่ไม่เกี่ยวข้อง

> **ไดเรกทอรี `commands/` แบบเดิม**: ปลั๊กอินบางตัวมีไดเรกทอรี `commands/` UI ของ Cowork ตอนนี้แสดงทั้ง commands และ skills เป็นหมวดหมู่ "Skills" เดียวกัน ดังนั้นให้จัดการไฟล์ `commands/*.md` เหมือนกับไฟล์ `skills/*/SKILL.md` เมื่อปรับแต่ง

**3. การปรับแต่งทั่วไป** — ไม่มี placeholder `~~` และผู้ใช้ต้องการแก้ไขปลั๊กอินโดยรวม อ่านไฟล์ปลั๊กอินเพื่อทำความเข้าใจการกำหนดค่าปัจจุบัน จากนั้นถามผู้ใช้ว่าต้องการเปลี่ยนแปลงอะไร

> **สำคัญ**: ห้ามเปลี่ยนชื่อปลั๊กอินหรือ skill ที่กำลังปรับแต่ง ห้ามเปลี่ยนชื่อไดเรกทอรี ไฟล์ หรือฟิลด์ชื่อปลั๊กอิน/skill

> **ผลลัพธ์ที่ไม่เป็นเทคนิค**: ผลลัพธ์ที่แสดงต่อผู้ใช้ทั้งหมด (รายการสิ่งที่ต้องทำ คำถาม สรุป) ต้องเขียนด้วยภาษาที่เข้าใจง่าย ไม่ใช้ศัพท์เทคนิค ห้ามกล่าวถึง prefix `~~`, placeholder หรือจุดปรับแต่งกับผู้ใช้ อธิบายทุกอย่างในแง่ของความสามารถของปลั๊กอินและเครื่องมือขององค์กร

## ขั้นตอนการปรับแต่ง

### ขั้นที่ 0: รวบรวมความต้องการของผู้ใช้ (เฉพาะการปรับแต่งเฉพาะส่วนและทั่วไป)

สำหรับ **การปรับแต่งเฉพาะส่วน** และ **การปรับแต่งทั่วไป** (ไม่ใช่การตั้งค่าปลั๊กอินทั่วไป) ตรวจสอบว่าผู้ใช้ให้บริบทเพิ่มเติมพร้อมกับคำขอหรือไม่ (เช่น "ปรับแต่ง skill standup — เราทำ standup แบบ async ใน #eng-updates ทุกเช้า")

- **หากผู้ใช้ให้บริบทแล้ว**: บันทึกไว้และใช้เพื่อเติมคำตอบล่วงหน้าในขั้นที่ 3 — ข้ามคำถามที่ผู้ใช้ตอบไปแล้ว
- **หากผู้ใช้ไม่ได้ให้บริบท**: ถามคำถามปลายเปิดหนึ่งข้อโดยใช้ AskUserQuestion ก่อนดำเนินการต่อ ปรับคำถามให้ตรงกับสิ่งที่ขอปรับแต่ง เช่น "คุณต้องการเปลี่ยนแปลงอะไรใน skill brief?" หรือ "คุณต้องการเปลี่ยนแปลงการทำงานของปลั๊กอินนี้อย่างไร?" ให้กระชับและตรงประเด็น

  ใช้คำตอบ (ถ้ามี) เป็นบริบทเพิ่มเติมตลอดขั้นตอนที่เหลือ

### ขั้นที่ 1: รวบรวมบริบทจาก Knowledge MCP

ใช้ Knowledge MCP ภายในองค์กรเพื่อรวบรวมข้อมูลที่เกี่ยวข้องกับขอบเขตการปรับแต่ง ดูรายละเอียดรูปแบบการค้นหาตามหมวดหมู่ที่ `references/search-strategies.md`

**สิ่งที่ต้องรวบรวม** (จำกัดเฉพาะที่เกี่ยวข้อง):

- ชื่อเครื่องมือและบริการที่องค์กรใช้
- กระบวนการและเวิร์กโฟลว์ขององค์กร
- แบบแผนของทีม (การตั้งชื่อ สถานะ มาตรวัดการประมาณ)
- ค่าการกำหนดค่า (ID เวิร์กสเปซ ชื่อโปรเจกต์ ตัวระบุทีม)

**แหล่งที่ค้นหา:**

1. **Chat/Slack MCP** — การกล่าวถึงเครื่องมือ การเชื่อมต่อ การอภิปรายเวิร์กโฟลว์
2. **Document MCP** — เอกสาร onboarding คู่มือเครื่องมือ คำแนะนำการตั้งค่า
3. **Email MCP** — การแจ้งเตือนใบอนุญาต อีเมลผู้ดูแล คำเชิญการตั้งค่า

บันทึกผลการค้นพบทั้งหมดเพื่อใช้ในขั้นที่ 3

### ขั้นที่ 2: สร้างรายการสิ่งที่ต้องทำ

สร้างรายการสิ่งที่ต้องทำเพื่อเปลี่ยนแปลง โดยกำหนดขอบเขตอย่างเหมาะสม:

- **สำหรับการปรับแต่งเฉพาะส่วน**: รวมเฉพาะรายการที่เกี่ยวข้องกับส่วนที่ผู้ใช้ขอ
- **สำหรับการตั้งค่าปลั๊กอินทั่วไป**: รัน `grep -rn '~~\w' /path/to/plugin --include='*.md' --include='*.json'` เพื่อค้นหาจุดปรับแต่ง placeholder ทั้งหมด จัดกลุ่มตามธีม
- **สำหรับการปรับแต่งทั่วไป**: อ่านไฟล์ปลั๊กอิน ทำความเข้าใจการกำหนดค่าปัจจุบัน และตามคำขอของผู้ใช้ ระบุสิ่งที่ต้องเปลี่ยน

ใช้คำอธิบายที่เข้าใจง่ายซึ่งมุ่งเน้นวัตถุประสงค์ของปลั๊กอิน:

- **ดี**: "เรียนรู้วิธีเตรียม standup ที่บริษัท"
- **ไม่ดี**: "แทนที่ placeholder ใน skills/standup-prep/SKILL.md"

### ขั้นที่ 3: ดำเนินการตามรายการ

ดำเนินการแต่ละรายการโดยใช้บริบทจากขั้นที่ 0 และขั้นที่ 1

**หากข้อมูลจากผู้ใช้ (ขั้นที่ 0) หรือ Knowledge MCP (ขั้นที่ 1) ให้คำตอบที่ชัดเจน**: ใช้โดยตรงโดยไม่ต้องขอยืนยัน

**มิฉะนั้น**: ใช้ AskUserQuestion ไม่ควรสันนิษฐานว่าค่าเริ่มต้น "มาตรฐานอุตสาหกรรม" ถูกต้อง — หากทั้งข้อมูลผู้ใช้และ Knowledge MCP ไม่ได้ให้คำตอบเฉพาะเจาะจง ให้ถาม หมายเหตุ: AskUserQuestion จะมีปุ่มข้ามและช่องข้อความอิสระเสมอ ดังนั้นไม่ต้องเพิ่มตัวเลือก "ไม่มี" หรือ "อื่นๆ"

**ประเภทการเปลี่ยนแปลง:**

1. **การแทนที่ placeholder** (การตั้งค่าทั่วไป): `~~Jira` → `Asana`, `~~your-org-channel` → `#engineering`
2. **การอัปเดตเนื้อหา**: แก้ไขคำสั่ง skill เวิร์กโฟลว์ หรือเอกสารอ้างอิงให้ตรงกับองค์กร
3. **การอัปเดตรูปแบบ URL**: `tickets.example.com/your-team/123` → `app.asana.com/0/PROJECT_ID/TASK_ID`
4. **ค่าการกำหนดค่า**: ID เวิร์กสเปซ ชื่อโปรเจกต์ ตัวระบุทีม

หากผู้ใช้ไม่ทราบหรือข้าม ให้คงค่าเดิมไว้ (หรือ placeholder ที่ขึ้นต้นด้วย `~~` สำหรับการตั้งค่าทั่วไป)

### ขั้นที่ 4: ค้นหา MCP ที่เป็นประโยชน์

หลังจากแก้ไขรายการปรับแต่งแล้ว เชื่อมต่อ MCP สำหรับเครื่องมือที่ระบุหรือเปลี่ยนแปลง ดูรายละเอียดขั้นตอน การแมปหมวดหมู่-คีย์เวิร์ด และรูปแบบไฟล์กำหนดค่าที่ `references/mcp-servers.md`

สำหรับเครื่องมือแต่ละตัวที่ระบุระหว่างการปรับแต่ง:

1. ค้นหาใน registry: `search_mcp_registry(keywords=[...])` โดยใช้คีย์เวิร์ดหมวดหมู่จาก `references/mcp-servers.md` หรือค้นหาชื่อเครื่องมือเฉพาะหากทราบแล้ว
2. หากยังไม่ได้เชื่อมต่อ: `suggest_connectors(directoryUuids=["chosen-uuid"])` — ผู้ใช้ดำเนินการ auth เอง
3. อัปเดตไฟล์กำหนดค่า MCP ของปลั๊กอิน (ตรวจสอบ `plugin.json` สำหรับตำแหน่งที่กำหนดเอง มิฉะนั้นใช้ `.mcp.json` ที่ root)

รวบรวมผล MCP ทั้งหมดและนำเสนอรวมกันในผลลัพธ์สรุป (ดูด้านล่าง) — ไม่ต้องนำเสนอ MCP ทีละตัวระหว่างขั้นตอนนี้

## การแพ็กเกจปลั๊กอิน

หลังจากปรับแต่งทั้งหมดเสร็จ แพ็กเกจปลั๊กอินเป็นไฟล์ `.plugin` สำหรับผู้ใช้:

1. **Zip ไดเรกทอรีปลั๊กอิน** (ไม่รวม `setup/` เนื่องจากไม่ต้องใช้แล้ว):
   ```bash
   cd /path/to/plugin && zip -r /tmp/plugin-name.plugin . -x "setup/*" && cp /tmp/plugin-name.plugin /path/to/outputs/plugin-name.plugin
   ```
2. **นำเสนอไฟล์ให้ผู้ใช้** ด้วยนามสกุล `.plugin` เพื่อให้สามารถติดตั้งได้โดยตรง (การนำเสนอไฟล์ .plugin จะแสดงให้ผู้ใช้เห็นเป็นตัวอย่างแบบ rich preview ที่สามารถดูไฟล์ปลั๊กอินทั้งหมด และยอมรับการปรับแต่งโดยกดปุ่ม)

> **สำคัญ**: สร้าง zip ใน `/tmp/` ก่อนเสมอ จากนั้นคัดลอกไปยังโฟลเดอร์ outputs การเขียนตรงไปยังโฟลเดอร์ outputs อาจล้มเหลวเนื่องจากสิทธิ์และทิ้งไฟล์ชั่วคราว

> **การตั้งชื่อ**: ใช้ชื่อไดเรกทอรีปลั๊กอินเดิมสำหรับไฟล์ `.plugin` (เช่น ถ้าไดเรกทอรีปลั๊กอินคือ `coder` ไฟล์ output ควรเป็น `coder.plugin`) ห้ามเปลี่ยนชื่อปลั๊กอินหรือไฟล์ระหว่างการปรับแต่ง — แทนที่เฉพาะค่า placeholder และอัปเดตเนื้อหา

## ผลลัพธ์สรุป

หลังจากปรับแต่ง นำเสนอสรุปสิ่งที่เรียนรู้แบ่งตามแหล่งที่มาให้ผู้ใช้ รวมส่วน MCP เสมอที่แสดงว่า MCP ใดเชื่อมต่อระหว่างการตั้งค่าและตัวใดที่ผู้ใช้ยังควรเชื่อมต่อ:

```markdown
## จากการค้นหา Slack

- คุณใช้ Asana สำหรับการจัดการโปรเจกต์
- รอบสปรินต์ 2 สัปดาห์

## จากการค้นหาเอกสาร

- Story points ใช้ขนาดเสื้อยืด

## จากคำตอบของคุณ

- สถานะ ticket: Backlog, In Progress, In Review, Done
```

จากนั้นนำเสนอ MCP ที่เชื่อมต่อระหว่างการตั้งค่าและตัวที่ผู้ใช้ยังควรเชื่อมต่อ พร้อมคำแนะนำวิธีเชื่อมต่อ

หากไม่มี Knowledge MCP ในขั้นที่ 1 และผู้ใช้ต้องตอบคำถามด้วยตนเองอย่างน้อยหนึ่งข้อ ให้เพิ่มข้อความท้ายนี้:

> อนึ่ง การเชื่อมต่อแหล่งข้อมูลอย่าง Slack หรือ Microsoft Teams จะช่วยให้ค้นหาคำตอบได้โดยอัตโนมัติในครั้งถัดไปที่คุณปรับแต่งปลั๊กอิน
